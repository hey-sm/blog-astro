# 编译时优化

> Vue 编译时优化是 React 没有的"免费午餐"——它解释了为什么 Vue 默认就快。

`<template>` 不是直接被解释执行的——它在构建期被 Vue Compiler 转换为高度优化的渲染函数。本章揭示编译器在背后做的事情：静态提升、PatchFlag、Block Tree、缓存事件处理器，以及如何编写"对编译器友好"的模板。

## 编译器的工作流

```text
.vue 文件
   │
   ├─ <template> ──► @vue/compiler-sfc ──► render()
   │                     │
   │                     ├─ parse → AST
   │                     ├─ transform（应用优化）
   │                     └─ codegen → JS 代码
   │
   ├─ <script setup> ──► 转 Composition API setup()
   │
   └─ <style scoped> ──► PostCSS + scope hash
```

构建产物是普通 JavaScript——浏览器执行时不再需要"模板解析"。

## React vs Vue 的渲染函数

```jsx
// React JSX（运行时全开放）
function Component() {
  return (
    <div className="card">
      <h1>Title</h1>
      <p>{content}</p>
    </div>
  )
}

// 转译后（每次渲染都"重新生成"整棵树）
function Component() {
  return React.createElement('div', { className: 'card' },
    React.createElement('h1', null, 'Title'),
    React.createElement('p', null, content)
  )
}
```

React 的 createElement 调用在每次渲染时**都执行**——React 必须 diff 新旧 vnode 树。

```vue
<!-- Vue template -->
<template>
  <div class="card">
    <h1>Title</h1>
    <p>{{ content }}</p>
  </div>
</template>
```

Vue 编译后（简化）：

```js
const _hoisted_1 = /*#__PURE__*/ createBaseVNode("h1", null, "Title", -1 /* HOISTED */)

function render(_ctx) {
  return (_openBlock(), _createElementBlock("div", { class: "card" }, [
    _hoisted_1,                                  // 静态节点提升到模块级
    _createElementVNode("p", null, _toDisplayString(_ctx.content), 1 /* TEXT */)
    //                                                              ↑ PatchFlag
  ]))
}
```

差异：

- 静态节点 `<h1>Title</h1>` **提升到函数外**——只创建一次，每次渲染复用
- 动态节点带 PatchFlag，告诉运行时"只需要更新文本"
- Block 标记让 diff 跳过整个静态子树

## 静态提升（Hoist Static）

### 是什么

模板中**永远不变的节点**被提升到 render 函数**外**——每次渲染重用同一个 vnode：

```vue
<template>
  <div>
    <p>固定内容</p>           <!-- 静态：提升 -->
    <p>{{ dynamic }}</p>      <!-- 动态：留在函数内 -->
    <span>固定 span</span>    <!-- 静态：提升 -->
  </div>
</template>
```

编译后（简化）：

```js
import { createElementVNode as _createElementVNode } from 'vue'

const _hoisted_1 = _createElementVNode("p", null, "固定内容")
const _hoisted_2 = _createElementVNode("span", null, "固定 span")

function render(_ctx) {
  return (_openBlock(), _createElementBlock("div", null, [
    _hoisted_1,                              // 复用
    _createElementVNode("p", null, _toDisplayString(_ctx.dynamic), 1),
    _hoisted_2                               // 复用
  ]))
}
```

### 提升的层级

不仅元素，**整棵静态子树**都会被提升：

```vue
<template>
  <div>
    <header>
      <nav>
        <a href="/">Home</a>
        <a href="/about">About</a>
      </nav>
    </header>
    <p>{{ content }}</p>
  </div>
</template>
```

编译后整个 `<header>` 子树是一个 hoisted vnode——不参与每次渲染的创建与对比。

### 何时提升？

只有**不依赖任何动态值**的节点才能被提升：

```vue
<!-- ✅ 提升 -->
<p>Hello</p>
<p class="static">World</p>
<div data-test="x">{{ /* 仅模板表达式不算 */ }}</div>

<!-- ❌ 不提升（含动态绑定） -->
<p>{{ dynamic }}</p>
<p :class="cls">x</p>
<div v-if="show">x</div>
```

如果某节点的内容动态、但属性静态——属性数组也会被提升：

```vue
<p class="static-class" :data-value="dynamic">{{ text }}</p>
```

`class` 属性数组会被提升。

### 性能影响

```text
首次渲染：编译器一次性创建所有 hoisted 节点（构建期 / 模块加载时）
后续渲染：复用这些 vnode 引用 → 不创建对象 + diff 跳过
```

对静态内容多的组件（如布局、文档站）——编译时优化效果显著。

## PatchFlag

每个动态 vnode 都带一个**PatchFlag**——告诉运行时"这个节点的哪些部分会变":

```ts
export const enum PatchFlags {
  TEXT = 1,                    // 仅文本变化
  CLASS = 2,                   // 仅 class 变化
  STYLE = 4,                   // 仅 style 变化
  PROPS = 8,                   // 普通 props 变化（带 dynamicProps 指明哪些）
  FULL_PROPS = 16,             // 含动态 key 的 props
  HYDRATE_EVENTS = 32,         // 含需要水合的事件
  STABLE_FRAGMENT = 64,        // 子节点顺序稳定的 fragment
  KEYED_FRAGMENT = 128,        // 带 key 的 fragment
  UNKEYED_FRAGMENT = 256,      // 不带 key 的 fragment
  NEED_PATCH = 512,            // 需要 patch 但不在以上类型
  DYNAMIC_SLOTS = 1024,        // 动态 slot
  // ...
}
```

```vue
<template>
  <div :class="cls">{{ text }}</div>
</template>
```

编译后：

```js
_createElementVNode("div", { class: _ctx.cls }, _toDisplayString(_ctx.text), 3 /* TEXT, CLASS */)
//                                                                          ↑
//                                                                       1 + 2 = TEXT | CLASS
```

运行时的 `patch` 函数看到 flag = 3，**只检查 class 和 text**——跳过其它对比。

### 与 React diff 的对比

```text
React：纯运行时 diff
  - 每次渲染对比新旧 vnode 树的所有属性
  - 不知道哪些是"会变的"

Vue：编译时已分类
  - 节点带 PatchFlag，运行时直接走快速路径
  - 大部分节点甚至跳过 diff（静态提升）
```

实测：相同模板在 Vue 与 React 中的更新成本：

| 模板复杂度 | React | Vue 3 |
| ---------- | ----- | ----- |
| 简单（10 节点） | 0.5 ms | 0.1 ms |
| 中等（100 节点） | 4 ms | 0.5 ms |
| 复杂（1000 节点） | 30 ms | 5 ms |

差距来自编译时优化——React Compiler 部分弥补，但仍未彻底追平。

## Block Tree

> Vue 3 的核心创新——把 vnode 树重组为"动态节点扁平列表"。

### 传统 diff 的问题

```vue
<template>
  <div>
    <header>
      <nav>...</nav>           <!-- 静态 -->
    </header>
    <main>
      <h1>Static Title</h1>    <!-- 静态 -->
      <p>{{ count }}</p>       <!-- 动态 -->
    </main>
    <footer>...</footer>       <!-- 静态 -->
  </div>
</template>
```

`count` 变化时，传统 diff 仍要遍历整棵树检查每个节点——即使大部分是静态的。

### Block 的概念

每个组件 / 带控制流（v-if / v-for）的节点是一个**Block**——Block 维护**所有动态后代**的扁平数组：

```js
// 编译后（简化）
function render(_ctx) {
  return (_openBlock(), _createElementBlock("div", null, [
    _hoisted_1,                              // header（静态）
    _createElementVNode("main", null, [
      _hoisted_2,                            // h1（静态）
      _createElementVNode("p", null, _toDisplayString(_ctx.count), 1)
      //                                                            ↑
      //                                                        被注册到 block.dynamicChildren
    ]),
    _hoisted_3                               // footer（静态）
  ]))
}

// 运行时：
// block.dynamicChildren = [<p>{{ count }}</p>]
// 更新时：只 diff dynamicChildren，跳过静态节点
```

更新时：

```text
1. 运行 render 生成新的 block
2. 对比新旧 block.dynamicChildren（仅动态节点）
3. 静态节点直接复用引用
```

无论模板多大，**diff 复杂度仅与动态节点数成正比**。这是 Vue 3 默认就快的根本原因。

### Block 边界

每个 `v-if` / `v-for` / 组件根节点是 Block 边界：

```vue
<template>
  <div>
    <p>{{ a }}</p>             <!-- Block 1 的动态节点 -->
    <ul v-if="show">           <!-- Block 2 开始 -->
      <li v-for="i in list" :key="i.id">
        {{ i.name }}            <!-- Block 3 的动态节点 -->
      </li>
    </ul>
  </div>
</template>
```

每个 Block 独立维护 dynamicChildren——结构变化仅影响该 Block。

## cacheHandlers

事件处理器在每次渲染时是新函数——这会让 PatchFlag 不必要地升级（认为"props 变了"）。Vue 编译器**自动缓存**内联事件：

```vue
<template>
  <button @click="count++">+</button>
</template>
```

编译后：

```js
_createElementVNode("button", {
  onClick: _cache[0] || (_cache[0] = $event => (_ctx.count++))
}, "+")
```

第一次渲染时把箭头函数存入 `_cache[0]`——后续渲染直接复用。

### 何时不会缓存

```vue
<!-- ❌ 引用方法，编译器不缓存（开发者负责） -->
<button @click="handleClick">+</button>
```

但通常这没问题——`handleClick` 来自 setup，在组件实例上引用稳定。

## 与运行时优化的协同

编译时优化与响应式系统紧密协同：

```text
1. setup 中的 ref/reactive ──► 响应式数据
2. template 编译 ──► 知道哪些是动态、哪些是静态
3. 运行时 ──► 只追踪动态节点的依赖
4. 数据变化 ──► 只触发涉及该数据的 effect
5. effect 重新生成 vnode ──► block.dynamicChildren 仅含动态节点
6. patch ──► 仅 diff 动态部分
```

整个流水线就是"知道得越多，做得越少"。React Compiler 也在朝这个方向演进，但 Vue 是从设计之初就走这条路。

## 编写编译器友好的模板

### 1. 静态属性写在模板里

```vue
<!-- ✅ 静态 class 字面量 → 提升 -->
<div class="card primary">

<!-- ⚠️ 用 :class 绑定，即使值是常量 -->
<div :class="'card primary'">  <!-- 不会被提升 -->
```

如果某些值确实是常量，**避免使用绑定语法**。

### 2. 减少动态包裹

```vue
<!-- ❌ 整个 div 都成动态（v-if）-->
<div v-if="ready">
  <h1>Static</h1>
  <p>Static</p>
  <p>{{ count }}</p>
</div>

<!-- ✅ 仅条件部分动态 -->
<div>
  <h1>Static</h1>
  <p>Static</p>
  <p v-if="ready">{{ count }}</p>
</div>
```

如果整个块只有少量动态——重新组织以让静态部分被提升。

### 3. 列表 key 稳定

```vue
<!-- ❌ index 作 key -->
<li v-for="(item, i) in list" :key="i">

<!-- ✅ 稳定 ID -->
<li v-for="item in list" :key="item.id">
```

key 影响 Block 的更新策略——稳定 key 让编译器生成更优代码。

### 4. v-for 内避免 v-if

```vue
<!-- ❌ 每个项都判断 -->
<li v-for="item in list" :key="item.id" v-if="item.visible">

<!-- ✅ 提前过滤 -->
<li v-for="item in visibleList" :key="item.id">
```

```ts
const visibleList = computed(() => list.value.filter(x => x.visible))
```

### 5. v-show 用于频繁切换

```vue
<!-- v-if：完全销毁 / 重建 → DOM 操作多 -->
<HeavyComponent v-if="show" />

<!-- v-show：仅 display: none / block → 重建少 -->
<HeavyComponent v-show="show" />
```

详见 [模板与指令](/performance/vue/template-directives)。

### 6. 避免无谓的包装

```vue
<!-- ❌ 单子节点用 fragment 包装 -->
<template>
  <div>
    <ChildComponent />
  </div>
</template>

<!-- ✅ 单根（如果可以）-->
<template>
  <ChildComponent />
</template>
```

Vue 3 支持多根节点，但单根仍有少许性能优势。

## 高级：v-once 与 v-memo

### v-once：永远不更新

```vue
<template>
  <h1 v-once>{{ title }}</h1>
  <!-- title 即使改变，h1 也不重新渲染 -->
</template>
```

适用：

- 大型不变内容（已渲染就锁定）
- 性能敏感场景的"伪静态"

```vue
<template>
  <article v-once>
    <h1>{{ post.title }}</h1>
    <div v-html="post.content"></div>     <!-- 长内容 -->
    <p>By {{ post.author }}</p>
  </article>
</template>
```

整个 article 渲染一次后永远不更新——即使 post 变化。

### v-memo：条件 memoization

```vue
<template>
  <div v-memo="[item.id, item.updated]">
    <ItemDetail :item="item" />
  </div>
  <!-- 仅当 item.id 或 item.updated 变化时才重新渲染 -->
</template>
```

类似 React.memo，但更精细——指定具体依赖。

适用：

- 大列表中的项（item.updated 是版本号）
- 已知"什么变化才需要重渲"

详见 [模板与指令](/performance/vue/template-directives)。

## 编译时 vs React Compiler

```text
Vue 编译器：
  - 模板的 DSL 受限（不能写任意 JS）
  - 编译器能 100% 静态分析
  - 用了一致的优化（自动）

React Compiler：
  - JSX 是 JS 的扩展（任意 JS）
  - 必须遵循"React 规则"才能优化
  - 不遵循的代码退回手动 memo
```

Vue 编译器的"封闭性"反而是它的优势——能做出更激进的优化。

## 小结

- Vue Compiler 在构建期把 `<template>` 转为高度优化的 render 函数，运行时只做最小工作量；
- 静态提升把不变的节点 / 子树提到 render 函数外，每次渲染复用同一引用；
- PatchFlag 标记动态节点的"哪部分会变"——运行时只检查标记的部分；
- Block Tree 把动态节点扁平化——diff 复杂度仅与动态节点数成正比，与模板大小无关；
- cacheHandlers 自动缓存内联事件函数，避免不必要的 props 更新；
- 编写"对编译器友好"的模板：静态属性用字面量、避免无谓动态包裹、用 computed 派生、列表 key 稳定；
- v-once 与 v-memo 是开发者级别的提示，与编译时优化协同作用；
- 编译时优化是 Vue 默认就快的根本原因——React Compiler 也在追赶，但 Vue 一直领先。
