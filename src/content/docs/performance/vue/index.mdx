Vue 的性能模型与 React 有本质差异：Vue 是**细粒度响应式**的——只有真正依赖的组件会重渲染，而 React 是**自顶向下重渲染**模型。这一差异让 Vue 在很多场景下"默认就快"，但也带来了 Vue 特有的性能陷阱（如响应式追踪过度）。

本子目录聚焦 Vue 3 + Composition API，覆盖响应式、编译、模板、SSR、虚拟化等全场景，最后落地到具体业务方案。

## Vue 性能模型的三个层次

```text
┌─ 1. 编译时层（Vue 独有的优势）─────────────┐
│   - 静态提升 (hoistStatic)                   │
│   - PatchFlag 标记动态部分                   │
│   - Block Tree 跳过静态子树                   │
│   - cacheHandlers 缓存事件处理器             │
│                                             │
│   开发者无感知，但收益巨大                    │
│   → 编译后只追踪"真正会变"的部分               │
└─────────────────────────────────────────────┘

┌─ 2. 响应式层（运行时调度）─────────────────┐
│   - Proxy 拦截 get/set                       │
│   - effect 收集依赖                          │
│   - flushJob 异步批处理                       │
│   - shallowRef / readonly 跳过深度追踪       │
│                                             │
│   性能问题多在此层                           │
│   → 大对象 / 嵌套数据 / 频繁更新              │
└─────────────────────────────────────────────┘

┌─ 3. 应用层（与 React 类似）────────────────┐
│   - 组件粒度（细 vs 粗）                      │
│   - 列表 key                                  │
│   - KeepAlive 缓存                           │
│   - 虚拟化、代码分割、懒加载                   │
└─────────────────────────────────────────────┘
```

理解这三层的分工，才能在出问题时知道**该往哪一层下手**。

## Vue 性能问题的常见类型

### 1. 响应式追踪过度

最常见的 Vue 特有问题。一个大的 `reactive` 对象，每个嵌套属性都被深度代理：

```ts
// ❌ 整个表格数据被深度代理
const tableData = reactive(largeArrayOfObjects)
// 每次访问 tableData[i].column 都触发 Proxy

// ✅ 浅层响应或冻结
const tableData = shallowRef(largeArrayOfObjects)
const frozenData = markRaw(largeArrayOfObjects)
```

详见 [浅层与只读优化](/performance/vue/shallow-optimization)。

### 2. 模板未利用编译优化

```html
<!-- ❌ 没用 v-once / v-memo 的不变内容 -->
<div>
  <h1>{{ title }}</h1>      <!-- 永不变 -->
  <ExpensiveList :items="list" />
</div>

<!-- ✅ 标记静态部分 -->
<div>
  <h1 v-once>{{ title }}</h1>
  <ExpensiveList :items="list" v-memo="[list.length]" />
</div>
```

详见 [模板与指令](/performance/vue/template-directives)。

### 3. computed / watch 滥用

```ts
// ❌ 每次访问都重新计算（不该用 computed）
const total = computed(() => Math.random())

// ❌ watch 触发链
watch(a, () => { b.value++ })
watch(b, () => { c.value++ })
watch(c, () => { ... })   // 难以追踪
```

详见 [computed / watch 调优](/performance/vue/computed-watch)。

### 4. 大列表无虚拟化

```html
<!-- ❌ 直接渲染 10000 行 -->
<div v-for="row in tableData" :key="row.id">
  <!-- 复杂模板 -->
</div>

<!-- ✅ 虚拟化 -->
<RecycleScroller :items="tableData" :item-size="50">
  <template #default="{ item }"><!-- ... --></template>
</RecycleScroller>
```

详见 [列表虚拟化](/performance/vue/virtualization)。

### 5. SSR 水合慢

Nuxt 应用在低端设备上的客户端水合可能 1-3 秒——优化路径包含 islands、selective hydration、payload 拆分。

详见 [Nuxt SSR / SSG / 水合](/performance/vue/ssr)。

## 快速诊断流程

```text
Vue 应用感觉卡？
│
├─ 操作输入卡顿（INP > 200 ms）?
│   ├─ Vue DevTools → Timeline 录制 → 找长任务
│   ├─ 是不是大对象用 reactive 而不是 shallowRef ?
│   ├─ computed 是否被频繁失效重算？
│   └─ 列表是否未虚拟化？
│
├─ 切页慢（路由切换 > 300 ms）?
│   ├─ 路由组件是否 defineAsyncComponent ?
│   ├─ KeepAlive 是否过度缓存（max 没设）?
│   └─ JS bundle 是否过大？
│
├─ 首屏 LCP 慢（> 2.5 s）?
│   ├─ Nuxt SSR 是否启用？
│   ├─ payload size 是否过大？
│   └─ Hydration 是否阻塞 LCP？
│
└─ 滚动 / 动画卡顿?
    ├─ Transition 是否走 transform/opacity ?
    ├─ v-for 是否用了正确的 key ?
    └─ 列表是否需要虚拟化？
```

## 阅读建议

- **响应式优先**：Vue 性能问题 70% 出在响应式追踪上。前三章是必读基础。
- **编译时是优势**：Vue 的编译时优化是 React 没有的"免费午餐"，理解它能解释很多"为什么 Vue 默认就快"。
- **Composition API 优先**：本系列代码以 Composition API + `<script setup>` 为主。Options API 用法在不同处会注明。
- **以 Vue 3.4+ 为基线**：Vue 3.4 引入了 Reactivity Transform 移除、defineModel、改进的 hydration 等，本系列以最新稳定版为参照。

## 与 React 的核心差异

```text
              React                          Vue 3
─────────────────────────────────────────────────────────────
渲染粒度     组件级（fiber 节点）             effect 级（单个属性）
默认行为     自顶向下重渲染                    精确依赖追踪
优化心智     "防止不必要渲染"                  "防止响应式过度追踪"
typical 工具  React.memo / useMemo            shallowRef / markRaw
内置缓存     ❌（手动 memoize）                computed 自动缓存
编译时优化   ⚠️（React Compiler 兴起）         ✅ 一直是核心
```

不要把 React 心智直接套用到 Vue——会写出"看起来正确但完全没用"的优化代码。
