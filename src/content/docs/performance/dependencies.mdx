# 依赖管理

> "Keep your dependencies up to date" / "Keep an eye on the size of dependencies"（roadmap 条目）

现代前端项目动辄 100+ 个 npm 依赖，bundle 体积的 70-90% 通常来自第三方代码。**选错一个库可能让 bundle 多 200 KB**，而开发者完全没意识到。本章覆盖如何评估依赖体积、选择 Tree-shaking 友好的库、监控 bundle 增长、寻找轻量替代方案，以及保持依赖更新的工程化方法。

## 依赖的隐性成本

```text
npm install some-package
   │
   ├─ 体积成本
   │   ├─ bundle 增长 X KB
   │   ├─ 影响 Tree Shaking 的副作用
   │   └─ 依赖树中的传递依赖（5 个包 → 实际 50 个）
   │
   ├─ 性能成本
   │   ├─ 解析 / 编译时间
   │   ├─ 运行时开销（如 polyfill 注入）
   │   └─ 主线程阻塞
   │
   ├─ 维护成本
   │   ├─ 安全更新
   │   ├─ Breaking change 升级
   │   └─ 版本冲突
   │
   └─ 信任成本
       ├─ 维护者是否活跃
       ├─ 是否被收购 / 弃坑
       └─ 供应链攻击风险
```

> 依赖不是免费的——加包前必须问"是否值得"。

## 评估包体积

### Bundlephobia

[Bundlephobia](https://bundlephobia.com) 是查询 npm 包体积的标准工具：

```text
访问：https://bundlephobia.com/package/lodash
显示：
  Bundle size:  72.4 KB minified
                25.7 KB minified + gzipped
  Composition:  Pure JS
  Has side effects: No
  Tree-shakeable: ⚠️ Mostly
  Download time on Slow 3G: 0.5 s
```

适合**装包前**检查体积。可以用浏览器扩展 [`import-cost`](https://marketplace.visualstudio.com/items?itemName=wix.vscode-import-cost) 在 VS Code 中实时显示导入成本：

```ts
import _ from 'lodash'        // 70.6KB (gzipped: 24.3KB) ⚠️
import { debounce } from 'lodash'  // 70.6KB (gzipped: 24.3KB) ⚠️ ← 整包
import debounce from 'lodash/debounce'  // 1.1KB (gzipped: 524B) ✅
```

### `pkg-size`

```bash
npx pkg-size moment
```

显示真实磁盘占用 + bundle 后体积。

### `bundlejs.com`

在线工具，输入 import 语句即时显示打包后体积：

```text
import { format } from 'date-fns'
→ 2.1 KB (gzipped)
```

## Tree-shaking 友好度

### ESM vs CJS

```js
// ❌ CJS：无法静态分析，整包打入
const _ = require('lodash')

// ✅ ESM：静态导入，按需打包
import { debounce } from 'lodash-es'
```

判断库是否提供 ESM：

```bash
# 看 package.json 的 main / module / exports
cat node_modules/lodash/package.json | grep -E "main|module|exports"
```

### 现代库的双格式

```json
{
  "main": "./dist/index.cjs.js",        ← Node.js / 老打包器
  "module": "./dist/index.esm.js",      ← 现代打包器
  "exports": {                          ← 最现代（推荐）
    ".": {
      "import": "./dist/index.esm.js",
      "require": "./dist/index.cjs.js",
      "types": "./dist/index.d.ts"
    }
  }
}
```

`exports` 字段让打包器精确选择正确版本。

### `sideEffects` 标记

```json
{
  "name": "my-lib",
  "sideEffects": false             ← 整个包无副作用，可放心摇
}
```

或针对性：

```json
{
  "sideEffects": [
    "*.css",
    "./src/polyfills.js"
  ]
}
```

错误的 `sideEffects` 配置会让打包器**保留全部代码**——有时一个不对的 `sideEffects: true` 会让 bundle 大 200%。

### 检测 Tree Shaking 是否生效

```js
// utils.js
export function used() {}
export function unused() {}

// main.js
import { used } from './utils'
used()
```

构建后用 `visualizer` 看是否 `unused` 不在 bundle 里。

## 常见库的轻量替代

### lodash → 按需 / lodash-es / Native

```js
// ❌ 70 KB
import _ from 'lodash'
_.debounce(fn, 100)

// ✅ 1 KB（按文件）
import debounce from 'lodash/debounce'

// ✅ ESM 版本，Tree-shaking 友好
import { debounce } from 'lodash-es'

// ✅✅ 现代浏览器原生 + 自实现
function debounce(fn, ms) {
  let timer
  return (...args) => {
    clearTimeout(timer)
    timer = setTimeout(() => fn.apply(this, args), ms)
  }
}
```

很多 lodash 方法在 ES2015+ 已有原生替代：

| Lodash | 原生 |
| ------ | ---- |
| `_.map` | `Array.prototype.map` |
| `_.filter` | `Array.prototype.filter` |
| `_.find` | `Array.prototype.find` |
| `_.cloneDeep` | `structuredClone()` |
| `_.isArray` | `Array.isArray` |
| `_.uniq` | `[...new Set(arr)]` |
| `_.merge` | `{ ...a, ...b }` 或 `Object.assign` |

### moment → date-fns / day.js / 原生 Intl

```js
// ❌ Moment.js 整包 + locale ≈ 230 KB
import moment from 'moment'
moment().format('YYYY-MM-DD')

// ✅ date-fns 按需 ≈ 几 KB
import { format } from 'date-fns'
format(new Date(), 'yyyy-MM-dd')

// ✅ Day.js + 插件 ≈ 7 KB
import dayjs from 'dayjs'
dayjs().format('YYYY-MM-DD')

// ✅✅ 原生 Intl ≈ 0 KB（浏览器内置）
new Intl.DateTimeFormat('zh-CN', { dateStyle: 'short' }).format(new Date())
```

> Moment.js 已被作者标记为"维护模式"，新项目不应使用。

### axios → fetch / ky / wretch

```js
// ❌ axios 整包 ≈ 15 KB
import axios from 'axios'
const data = await axios.get('/api/users').then(r => r.data)

// ✅ 原生 fetch ≈ 0 KB
const data = await fetch('/api/users').then(r => r.json())

// ✅ ky（基于 fetch 的轻封装）≈ 4 KB
import ky from 'ky'
const data = await ky.get('/api/users').json()
```

### immutable.js → immer

```js
// ❌ immutable.js ≈ 60 KB
import { Map, List } from 'immutable'

// ✅ immer ≈ 7 KB
import { produce } from 'immer'
const next = produce(state, draft => {
  draft.user.name = 'new'
})
```

### query-string → URLSearchParams

```js
// ❌ qs ≈ 10 KB
import qs from 'qs'
qs.stringify({ a: 1, b: 2 })

// ✅ 原生
new URLSearchParams({ a: 1, b: 2 }).toString()
```

### uuid → crypto.randomUUID

```js
// ❌ uuid v4 ≈ 5 KB
import { v4 as uuidv4 } from 'uuid'
uuidv4()

// ✅ 现代浏览器 + Node 19+ ≈ 0 KB
crypto.randomUUID()
```

### classnames → clsx

```js
// classnames ≈ 800 B
import classNames from 'classnames'

// clsx 是 classnames 的更小替代 ≈ 400 B（API 兼容）
import clsx from 'clsx'
```

### React Icons → lucide-react / heroicons

```jsx
// ❌ react-icons：默认导入会进很多图标
import { FiHome } from 'react-icons/fi'
// 每个图标按 ~ 1 KB，但 Tree Shaking 不彻底

// ✅ lucide-react：每个图标精准 Tree-shake
import { Home } from 'lucide-react'
```

## 第三方 UI 库的体积

主流组件库的体积参考：

| 库 | 全包大小 | 按需后甜点 | 备注 |
| -- | -------- | ---------- | ---- |
| Material UI v5 | ~ 350 KB gzip | 按组件 5-15 KB | 有 emotion 运行时 |
| Ant Design | ~ 600 KB gzip | 按组件 5-20 KB | dayjs 替代 moment 后变小 |
| Chakra UI | ~ 200 KB gzip | 按组件 3-10 KB | 主题运行时 |
| Radix UI Primitives | ~ 0 + 按组件 | 5-15 KB / 组件 | 无样式，最小 |
| shadcn/ui | 拷贝代码 | 仅你用的部分 | 不是 npm 包 |
| Headless UI | ~ 30 KB | 按组件 5-10 KB | 无样式 |

**趋势**：从"重型组件库"向"原子化、可拷贝、无样式 primitives"演进。

### 按需导入

#### Ant Design

```js
// ❌ 整包导入（60+ MB 源码）
import { Button } from 'antd'

// ✅ Vite 自动按需（无需配置）
// 实际只打包 Button + 依赖
```

或用 `babel-plugin-import`：

```js
{
  plugins: [
    ['import', { libraryName: 'antd', style: 'css' }]
  ]
}
```

#### MUI

```js
// ❌ 大量编译时间 + bundle
import { Button } from '@mui/material'

// ✅ 直接路径，编译更快
import Button from '@mui/material/Button'
```

## 监控 bundle 增长

### CI 中检查 bundle 体积

```yaml
# .github/workflows/bundle-size.yml
name: Bundle Size
on: pull_request
jobs:
  size:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci && npm run build
      - uses: andresz1/size-limit-action@v1
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
```

`size-limit` 配置：

```json
{
  "size-limit": [
    {
      "path": "dist/assets/main.*.js",
      "limit": "200 KB"
    },
    {
      "path": "dist/assets/vendor.*.js",
      "limit": "150 KB"
    }
  ]
}
```

PR 中超额会自动拦截，且评论显示具体增量：

```text
Size limit ⚠️
| Path             | Size      | Limit  |
| ---------------- | --------- | ------ |
| dist/main.*.js   | 215 KB ↑  | 200 KB |
                                ❌ 超出 15 KB
```

### Bundlewatch / `bundlemon`

类似工具，配 GitHub Status Check。

### 自动评论 PR 的 bundle 变化

```yaml
- uses: preactjs/compressed-size-action@v2
  with:
    repo-token: '${{ secrets.GITHUB_TOKEN }}'
```

PR 评论显示每个 chunk 的变化（gzip / brotli），是否新增、修改：

```text
File                | Before    | After     | Change
main.abc.js         | 145.3 KB  | 152.1 KB  | +6.8 KB ↑
vendor.def.js       | 98.4 KB   | 98.4 KB   | =
```

## 保持依赖更新

> "Keep your dependencies up to date" 不仅是安全建议，更是性能要求。

### 为什么要保持更新

- **性能**：库作者持续优化（React 18 → 19 性能持续提升）
- **bundle 体积**：旧版常带累赘的 polyfill / 工具方法
- **安全**：CVE 修复
- **新特性**：可能消除你正在 workaround 的痛点

### 工具

#### `npm outdated` / `pnpm outdated`

```bash
$ pnpm outdated
Package      Current  Wanted   Latest
react         18.2.0  18.2.0   19.0.0
typescript    5.0.4   5.0.4    5.5.4
vite          4.0.0   4.0.5    5.4.0
```

#### `npm-check-updates`

```bash
npx npm-check-updates -u    # 升级 package.json 中所有版本
npm install
```

激进策略：每周或每月跑一次。

#### Renovate / Dependabot

GitHub 应用，自动创建升级 PR：

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: 'npm'
    directory: '/'
    schedule:
      interval: 'weekly'
    groups:
      minor-and-patch:
        update-types: ['minor', 'patch']
```

每周自动提 PR 升级 minor / patch。Major 版本单独 PR（需人工评估 breaking change）。

#### `taze` (pnpm 友好)

```bash
npx taze -w
```

更适合 monorepo / pnpm workspace。

### 升级策略

```text
patch (x.y.Z)：✅ 几乎无脑升
minor (x.Y.z)：✅ 小心，看 changelog
major (X.y.z)：⚠️ 需评估 breaking change
```

实践：

- patch / minor → 自动 PR + CI 通过自动合
- major → 人工 PR，配迁移指南

### lock 文件的重要性

```text
package-lock.json / pnpm-lock.yaml / yarn.lock
```

保证全员 / CI / 生产用**完全相同的依赖版本**。绝不要 `.gitignore` 它们。

```bash
# 确保安装严格按 lock
npm ci                # 比 install 严格，CI 中用
pnpm install --frozen-lockfile
```

## 依赖审计

### 审计工具

```bash
# 安全漏洞
npm audit
pnpm audit

# 重复依赖（不同版本同时存在）
npm dedupe
pnpm dedupe

# 未使用的依赖
npx depcheck

# 解析整个依赖树
npm ls react        # 哪些包依赖了 react
pnpm why react
```

### `depcheck` 找未使用

```bash
$ npx depcheck
Unused dependencies:
  * lodash
  * moment
Missing dependencies:
  * @types/node: ./src/server.ts
```

清理后立刻减小 bundle。

### 重复依赖的危害

```text
react@18.2.0   ← your code 直接装
react@17.0.2   ← old-lib 的 peerDependency
```

两个版本的 React 同时存在 → bundle 翻倍 + 运行时错误（hooks 跨版本）。

```bash
# 看是否有重复
npm ls react
pnpm why react

# 强制去重
npm dedupe
pnpm dedupe
```

或在 `package.json` 中用 `overrides` / `resolutions` 强制版本：

```json
{
  "overrides": {
    "react": "18.3.1"
  }
}
```

## 锁定第三方代码

### `peerDependencies` 的正确处理

库作者应该把 React / Vue 这类**框架级依赖**声明为 peerDependencies：

```json
{
  "peerDependencies": {
    "react": "^18.0.0 || ^19.0.0"
  }
}
```

避免每个库都打包自己的 React 副本。

### `external` 配置

打包库时把 peer 依赖标记为 external：

```js
// vite.config.js（如果你在做库）
export default {
  build: {
    rollupOptions: {
      external: ['react', 'react-dom']
    }
  }
}
```

## 选型决策框架

加入新依赖前自问：

```text
1. 这个功能能用原生 API / 几行代码实现吗？
   └─ 能 ──► 不要装

2. 包大小是否合理？（用 Bundlephobia 查）
   └─ > 50 KB ──► 找替代或衡量必要性

3. 是否 Tree-shaking 友好？（ESM + sideEffects）
   └─ 否 ──► 优先级降低

4. 维护是否活跃？
   ├─ 最近 6 个月有更新吗？
   ├─ Issues 响应速度
   └─ 是否有迁移文档

5. 是否有同类轻量替代？
   └─ 用 'X alternative npm' 搜索

6. 是否会传染依赖？（看 dependencies）
   └─ 单包带入 20 个依赖时要警惕
```

### 经典案例：选 date-fns vs day.js

| 维度 | date-fns | day.js |
| ---- | -------- | ------ |
| 体积（按需） | 1-3 KB / 函数 | 7 KB（核心） |
| API 风格 | 函数式 | 链式（类似 moment） |
| Tree-shaking | 完美 | 有限（核心整包） |
| TypeScript | 好 | 好 |
| 国际化 | 优秀 | 需插件 |
| 不可变 | ✅ | ✅ |

**结论**：

- 用很多日期方法 → day.js（核心固定，加方法不增体积）
- 只用几个方法 → date-fns（按需打包更小）

## 反模式清单

| 反模式 | 后果 | 修复 |
| ------ | ---- | ---- |
| 装一个工具用一个方法 | bundle 浪费 | 自实现或找按需库 |
| `import _ from 'lodash'` | 70 KB 整包 | 按需路径 / lodash-es |
| 还在用 moment.js | 230 KB | 替换为 day.js / date-fns / Intl |
| 用 axios 但没用拦截器 | 15 KB 浪费 | 原生 fetch |
| 每次升级都跳一个 major | 一次升级痛苦 | renovate weekly |
| 不锁 lock 文件 | 全员版本不一致 | 提交 lock 文件 |
| 不审计 `depcheck` | 死代码累积 | 每季度审计一次 |
| 整个 UI 库打入 bundle | 几百 KB 浪费 | 按需导入 |

## 小结

- 加新依赖前先去 Bundlephobia 查体积，VSCode 用 import-cost 实时显示成本；
- 偏好 ESM + 正确 sideEffects 的库，按需导入 + Tree Shaking 是基本盘；
- lodash / moment / axios 等老牌大厂库都有更轻的现代替代，迁移可省百 KB 级别 bundle；
- 在 CI 中加 size-limit / bundlewatch，bundle 增长不应该悄无声息地发生；
- Renovate / Dependabot 自动 patch + minor 升级，major 单独评估，是健康的依赖更新节奏；
- 重复依赖（同包多版本）是 bundle 翻倍的隐性杀手，用 `pnpm why` / `npm ls` 定期审计；
- 选型决策不只看"能不能用"，还看体积 / Tree-shaking / 维护 / 替代方案 / 传染性；
- 写库时把框架声明为 peerDependencies，让用户的应用不会被打入多份框架代码。
