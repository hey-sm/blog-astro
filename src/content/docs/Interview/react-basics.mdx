# React 基础面试题

本文梳理了 React 面试基础篇的核心考察点，涵盖核心设计思想、JSX、生命周期演进、受控与非受控组件、类组件与函数式组件以及 props 与 state。

---

## 一、React 核心设计思想

- **组件化设计**：UI 分解为独立功能单元，通过 props 组合成组件树；HOC 与自定义 Hooks 实现逻辑复用
- **声明式编程**：`UI = f(state)`，开发者描述目标状态，框架处理渲染细节；相同 state 必定输出相同视图
- **高效更新引擎**：虚拟 DOM + 启发式 O(n) Diff + 批量更新 + Fiber 可中断渲染
- **单向数据流**：props 自上而下传递，严禁子级逆向修改；通过 Context/Redux 跨层级通信

这些原则使 React 具备高维护性（组件解耦）、高性能（智能更新）、可预测性（数据流透明）。

## 二、JSX

### JSX 的本质

JSX 是 JavaScript 的语法扩展，会被 Babel/TypeScript 编译为 `React.createElement()` 调用，生成虚拟 DOM 对象。

```jsx
// JSX 代码
const element = <div className="title">Hello React</div>

// 编译后
const element = React.createElement('div', { className: 'title' }, 'Hello React')

// React 17+ 新转换：通过 jsx-runtime 自动引入，无需手动 import React
import { jsx as _jsx } from 'react/jsx-runtime'
const element = _jsx('div', { className: 'title', children: 'Hello React' })
```

### JSX 与 HTML 的区别

- **属性命名**：小驼峰（`className`、`htmlFor`、`onClick`）
- **样式对象**：CSS 属性转小驼峰，`style={{ fontSize: 14 }}`
- **子元素类型**：支持字符串、JSX、数组；`false`/`null`/`undefined` 不渲染

### 表达式嵌入与条件渲染

```jsx
{/* 三元 */}
{isLoggedIn ? <UserPanel /> : <LoginButton />}

{/* 短路运算 */}
{isLoading && <Spinner />}

{/* 列表渲染：必须有 key */}
{items.map(item => <li key={item.id}>{item.name}</li>)}
```

### 安全特性

- **自动转义**：嵌入内容会被转义为字符串，防止 XSS
- **`dangerouslySetInnerHTML`**：显式插入原始 HTML（需手动防范风险）

### 高频面试题

- **为什么组件首字母必须大写？** React 通过首字母大小写区分原生 DOM 标签（`<div>`）和自定义组件（`<MyComponent>`）。
- **`<></>` 是什么？** `Fragment` 简写，包裹多个元素而不产生额外 DOM 节点。

## 三、生命周期方法演进

### 类组件生命周期

#### 挂载阶段（Mounting）

- `constructor`：初始化 state、绑定方法
- `getDerivedStateFromProps`：props 同步到 state
- `render`：生成虚拟 DOM
- `componentDidMount`：网络请求、DOM 操作、订阅事件

#### 更新阶段（Updating）

- `getDerivedStateFromProps`
- `shouldComponentUpdate`：返回 `false` 可阻止渲染（性能优化核心）
- `render`
- `getSnapshotBeforeUpdate`：获取 DOM 更新前状态（如滚动位置）
- `componentDidUpdate`

#### 卸载阶段（Unmounting）

- `componentWillUnmount`：清除定时器、取消订阅、释放资源

### 废弃的生命周期

| 废弃方法 | 替代方案 | 废弃原因 |
|---------|---------|---------|
| `componentWillMount` | `constructor` / `useEffect` | 异步渲染下可能多次执行 |
| `componentWillReceiveProps` | `getDerivedStateFromProps` | 易产生副作用与竞态 |
| `componentWillUpdate` | `getSnapshotBeforeUpdate` | 不安全副作用风险 |

⚠️ `componentWillXXX` 系列在并发模式（Concurrent Mode）下可能被多次调用，故标记为 `UNSAFE`。

### 函数组件用 useEffect 模拟

```js
useEffect(() => {
  const timer = setInterval(/* ... */)
  return () => clearInterval(timer) // componentWillUnmount
}, []) // [] → componentDidMount

useEffect(() => { /* ... */ })          // componentDidUpdate（任意更新）
useEffect(() => { /* ... */ }, [count]) // 特定依赖更新
```

### 错误边界

```jsx
class ErrorBoundary extends React.Component {
  state = { hasError: false }
  static getDerivedStateFromError(error) {
    return { hasError: true } // 渲染阶段：更新状态显示降级 UI
  }
  componentDidCatch(error, info) {
    logErrorToService(error, info) // 提交阶段：上报错误
  }
  render() {
    return this.state.hasError ? <FallbackUI /> : this.props.children
  }
}
```

## 四、受控组件与非受控组件

| 维度 | 受控组件 | 非受控组件 |
|------|---------|-----------|
| 数据存储 | React state | DOM 节点自身 |
| 更新机制 | `onChange` 同步更新 state | 用户输入直接改 DOM，用 `ref` 获取 |
| 验证时机 | 实时验证 | 提交时验证 |
| 文件上传 | 不支持 | **必须使用** |

```jsx
// 受控组件：value 绑定 state，onChange 同步
<input value={value} onChange={e => setValue(e.target.value)} />

// 非受控组件：ref 访问，defaultValue 设初始值
const inputRef = useRef(null)
<input defaultValue="初始值" ref={inputRef} />
```

### 关键问题

- **为什么文件输入必须用非受控组件？** 浏览器安全限制，JS 无法以编程方式设置 `<input type="file">` 的值。
- **如何避免受控组件 value 变成 undefined？** 确保 value 始终为字符串/数字，避免 `null`/`undefined`。
- **受控组件性能优化？** 高频输入用防抖（`debounce`），用 `React.memo` 隔离表单组件。

### 决策建议

需要实时验证/控制 → 受控；涉及文件上传或性能要求极高 → 非受控；默认推荐受控组件。

## 五、类组件 vs 函数式组件

| 维度 | 类组件 | 函数式组件 |
|------|--------|-----------|
| 定义方式 | ES6 类继承 `React.Component` | JavaScript 函数 |
| 状态管理 | `this.state` + `this.setState()` | `useState` / `useReducer` |
| 生命周期 | 完整生命周期方法 | `useEffect` 模拟 |
| 逻辑复用 | HOC | 自定义 Hooks |
| 代码量 | 冗长（需处理 this 绑定） | 简洁 |

### 两大经典陷阱

```js
// 1. this 绑定问题（类组件）
class Button extends React.Component {
  // ❌ 未绑定时 this 为 undefined
  // ✅ 方案：构造函数 bind 或箭头函数 handleClick = () => {}
}

// 2. 闭包陷阱（函数组件）
useEffect(() => {
  const timer = setInterval(() => setCount(count + 1), 1000) // ❌ 始终捕获初始值
  return () => clearInterval(timer)
}, [])

useEffect(() => {
  const timer = setInterval(() => setCount(prev => prev + 1), 1000) // ✅ 函数式更新
  return () => clearInterval(timer)
}, [])
```

### 现代趋势

- 官方推荐**函数式组件 + Hooks**，React 18 新特性（并发模式）优先支持 Hooks
- 类组件保留场景：旧项目维护、需要 Error Boundaries（`getDerivedStateFromError` / `componentDidCatch` 暂仅类组件支持）

## 六、props 与 state

| 维度 | props | state |
|------|-------|-------|
| 数据来源 | 外部传入（父 → 子） | 组件内部维护 |
| 可变性 | 只读（Immutable） | 可修改（`setState`/`useState`） |
| 作用范围 | 跨组件层级传递 | 组件私有 |
| 更新触发 | 父组件重渲染传新 props | 调用状态更新方法 |

### 数据流方向

单向数据流：props 只能自上而下传递，子组件通过**回调函数**触发父组件更新 state。

```jsx
function Parent() {
  const [count, setCount] = useState(0)
  return <Child count={count} onIncrement={() => setCount(c => c + 1)} />
}
```

### 常见陷阱

```js
// 1. props 不可变：禁止直接修改 props
// ❌ user.name = 'Bob'

// 2. state 异步更新
setCount(count + 1)
setCount(count + 1)          // ❌ 不会合并，只 +1
setCount(prev => prev + 1)
setCount(prev => prev + 1)   // ✅ 函数式更新，正确 +2
```

### 高频面试题

- **能否在子组件修改 props？** 不能，props 只读。需修改应通过父组件回调更新父级 state。
- **如何跨多层传递数据？** 用 Context API 或状态管理库（Redux/MobX），避免 props 逐层透传。
- **状态提升**：多个组件共享状态时，将 state 提升到最近的共同祖先。
