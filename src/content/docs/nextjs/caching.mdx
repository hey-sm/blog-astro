# 缓存体系

Next.js 16 的缓存从设计上分为四层。理解"什么缓存在哪里、何时失效"是写出高性能 Next.js 应用的前置条件。

## 四层缓存模型

```text
┌─────────────────────────────────────────────────────────────┐
│  1. Request Memoization（请求记忆化）                        │
│     同一 render 中相同 fetch 自动去重                        │
│     生命周期：单次 render                                   │
│     Persists: ❌ 跨请求不共享                                │
├─────────────────────────────────────────────────────────────┤
│  2. Data Cache（数据缓存）                                   │
│     fetch 返回值被持久化缓存，跨请求共享                     │
│     生命周期：永久（直到 revalidate / 手动失效）             │
│     Persists: ✓ 跨请求共享                                  │
├─────────────────────────────────────────────────────────────┤
│  3. Full Route Cache（全路由缓存）                           │
│     完整 HTML + RSC Payload 被缓存                           │
│     生命周期：永久（直到重新部署 / 手动失效）                │
│     Persists: ✓ 跨请求共享                                  │
├─────────────────────────────────────────────────────────────┤
│  4. Router Cache（客户端路由缓存）                           │
│     已访问的 RSC Payload 缓存在浏览器内存中                  │
│     生命周期：30s（或 session）                              │
│     Persists: ❌ 页面刷新丢失                                │
└─────────────────────────────────────────────────────────────┘
```

## 1. Request Memoization（请求记忆化）

React 的 `cache()` 和 Next.js 的 fetch 层在**同一个 render** 中自动去除重复：

```tsx
// 两个组件分别 fetch 同一个 URL
// 实际只发送一个 HTTP 请求

function ComponentA() {
  const data = await fetch('https://api.example.com/users')
  // ...
}

function ComponentB() {
  const data = await fetch('https://api.example.com/users')
  // 复用 ComponentA 的结果（同一个 render）
}
```

这只影响**同一 render** 内的重复请求——不同 render / 不同请求之间不共享。

### React `cache()` 函数

```tsx filename="lib/data.ts"
import { cache } from 'react'
import { db } from '@/lib/db'

const getPost = cache(async (slug: string) => {
  return db.post.findUnique({ where: { slug } })
})
```

`cache()` 记忆化**单个 render 中**相同参数的调用。render 结束后记忆清空。

## 2. Data Cache（数据缓存）

### fetch 层面

```tsx
// 默认：结果被持久化缓存
fetch('https://api.example.com/data')

// 禁用缓存
fetch('https://api.example.com/data', { cache: 'no-store' })

// 时间窗口缓存
fetch('https://api.example.com/data', { next: { revalidate: 60 } })
```

### `'use cache'` 层面 (Next.js 16)

```tsx
async function getFeaturedPosts() {
  'use cache'
  cacheTag('posts', 'featured')
  cacheLife({ revalidate: 3600 })

  return db.post.findMany({ where: { featured: true } })
}
```

### 对比：`cache()` vs `'use cache'`

| 维度 | `React.cache()` | `'use cache'` |
| ---- | --------------- | ------------- |
| 范围 | 单次 render | 跨请求 |
| 存储 | 内存 | 持久化缓存 |
| 失效 | 自动（render 结束） | 手动 `revalidateTag` / 时间到期 |
| 用途 | 去重（同 render） | 持久化缓存（跨请求） |

## 3. Full Route Cache（全路由缓存）

服务端渲染的完整 HTML + RSC Payload。对于静态路由和静态动态段，这个缓存产生在构建时。对于动态路由，**不会产生全路由缓存**（每次请求重新渲染）。

触发失效：
- `revalidateTag` / `revalidatePath`
- 重新部署
- `cookies.set` / `headers` 变化（即使路由级）

## 4. Router Cache（客户端路由缓存）

内存缓存，保存 RSC Payload。`<Link>` 预加载的页面也会写入此缓存。

### `staleTimes` 配置 (Next.js 16)

```ts filename="next.config.ts"
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  experimental: {
    staleTimes: {
      dynamic: 30,       // 动态路由的缓存有效期（秒）
      static: 300,       // 静态路由的缓存有效期（秒）
    },
  },
}

export default nextConfig
```

- `dynamic`: 浏览器中动态路由的 RSC Payload 在 30 秒内不会重新获取（非刷新）
- `static`: 静态路由的有效期为 5 分钟

你可以使用 `router.refresh()` 强制穿透客户端路由缓存。

## 缓存决策矩阵

```text
这个数据变化的有多频繁？
├─ 不常变（应用程序配置、导航、静态内容）
│   → 依赖 Data Cache / Full Route Cache
│   → 失效方式：revalidateTag / 重新部署
│
├─ 中等变化（用户内容、博文、产品列表）
│   → ISR + revalidate 时间窗口
│   → 失效方式：revalidate + 按需 revalidateTag
│
├─ 每次请求都不同（仪表盘、实时数据、cookies/h2drs 内：
│ → 不可能有数据缓存
│   → 使用 Router Cache 减少客户端重取
│
└─ 不缓存（敏感数据、临时结果）
    → `cache: 'no-store'` 或 `noStore()` 或动态 fetch
```

## 调试缓存行为

### 日志

```ts filename="next.config.ts"
const nextConfig: NextConfig = {
  logging: {
    fetches: {
      fullUrl: true,
    },
  },
}
```

构建日志中会显示 `(cache HIT)` 或 `(cache SKIP)`。

### 浏览器 Network 面板

- 静态路由 → HTML 从磁盘缓存
- 动态路由 → 每次请求都发出网络请求
- Router Cache → 路由导航无额外网络请求（RSC Payload 从内存）

## 陷阱

### 在开发模式下所有缓存默认关闭

`next dev` 中 Data Cache 和 Full Route Cache 均不生效。生产行为只能在 `next build && next start` 中观察。

### `noStore()` 与 `'use cache'` 互斥

```tsx
import { unstable_noStore } from 'next/cache'

async function getData() {
  unstable_noStore()                  // ← opt out 数据缓存
  'use cache'                         // ← opt in 数据缓存
  // 冲突 — 'use cache' 优先，unstable_noStore 无效
}
```

### 路由缓存不会跨 session

刷新页面 (CMD+R) → Router Cache 清空 → 所有 RSC Payload 重新从服务端获取。这无法配置（这是浏览器行为）。

### `<Link prefetch>` 在 dev 模式下不工作

prefetch 行为只在生产模式中生效。dev 模式下 `<Link>` 的行为与非 prefetch 相同。

## 小结

- 四层缓存从请求内到跨请求到客户端：Memoization → Data Cache → Full Route Cache → Router Cache；
- `React.cache()` 去重同 render 的调用，`'use cache'` 持久化跨请求的缓存；
- `staleTimes` 控制客户端路由缓存的 TTL；
- 静态路由尽量使用默认缓存，动态路由通过 `'use cache'` + ISR 获得类似效果；
- dev 模式下所有服务端缓存默认关闭，只相信 `next build && next start` 的结果。
